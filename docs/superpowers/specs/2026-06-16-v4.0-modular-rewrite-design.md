# Design: EU4 DLC Unlocker v4.0 — Modular Rewrite

**Date:** 2026-06-16
**Status:** Draft (pending user review)
**Supersedes:** v3.0 (CreamAPI 5.3.0.0, two monolithic scripts)

---

## 1. Goal

Rewrite the EU4 DLC Unlocker for macOS to fix all known defects (broken
translations, path-with-spaces bug, missing macOS Gatekeeper handling, missing
error handling, architecture misinformation, outdated DLC list maintenance)
and make it **stable, safe, and maintainable** long-term, while keeping a
clear path to extend it to other Paradox games without code changes.

This is a **modular architecture with only EU4 shipped**: the engine is
game-agnostic and reads per-game data files. Adding CK3/HOI4/Stellaris later
requires only a new `games/<game>.conf`, no engine edits.

## 2. Decisions (locked)

| Decision | Choice | Rationale |
|---|---|---|
| Scope | Modular engine, **EU4 only** shipped | Extensible without paying maintenance/risk cost upfront (YAGNI) |
| CreamAPI dylib storage | In git under `vendor/creamapi/` | Offline install (key for cs.rin.ru audience); CreamAPI is closed-source so no build option |
| DLC list strategy | `unlockall = true`, empty `[dlc]` | CreamAPI fetches DLC list from Steam servers; no manual list to maintain; future-proof for new DLC releases |
| CLI shape | Single `unlocker` entrypoint + back-compat wrappers | Removes duplication, one tested path, `status` subcommand, old script names still work |

### Evidence from the CreamAPI release archive

- The dylib currently in the repo is byte-identical to the official
  `nonlog_build/macos/libsteam_api.dylib` (SHA256
  `13f49f08eaf3e4f48103173605793c8da1bcaa04bf6f4a3adbd0b29a6ef5c0d9`).
- VirusTotal for that file: **0/64 detections** — clean, good counter to AV
  false positives.
- Changelog v5.0.0.0: *"Updated the logic of the 'unlockall' option. Now it
  should not conflict if the '[dlc]' section is not empty."* + v4.1.0.0:
  *"'[dlc]' section should NOT be filled when using this option."* → confirms
  `unlockall=true` + empty `[dlc]` is the correct, supported configuration.
  Requires the game to be online (Steam reachable) — acceptable since a
  legitimate base-game purchase is a stated requirement.
- Available hardening options for problem games: `extraprotection`,
  `filestohide` (documented, not enabled by default).

## 4. Architecture

### 4.1 Game-agnostic engine + data-driven games

The engine (`bin/lib/common.sh`) does **not** know which game it handles. It
reads a `games/<game>.conf` file that provides:

```ini
# games/eu4.conf
game_name = Europa Universalis IV
appid = 236850
app_bundle = eu4.app                 # where libsteam_api.dylib lives
search_paths =                       # one path per line; globs allowed
  /Users/$USER/Library/Application Support/Steam/steamapps/common/Europa Universalis IV
  /Volumes/*/SteamLibrary/steamapps/common/Europa Universalis IV
  /Volumes/*/Steam/steamapps/common/Europa Universalis IV
framework_subdir = Contents/Frameworks
# $USER is expanded by the engine at runtime.
```

To support a new Paradox game later: create `games/<game>.conf`. No engine
change. This is the modular-vs-universal resolution: universal architecture,
minimal shipped surface.

### 4.2 Single CLI: `bin/unlocker`

```
./unlocker install   [game]   # default game = eu4
./unlocker uninstall [game]
./unlocker status    [game]   # non-mutating: shows patch/backup state
./unlocker --help
./unlocker --version

Flags:  --yes (non-interactive)   --path <dir> (override search)
```

`install_dlc_unlocker.sh` and `uninstall_dlc_unlocker.sh` become thin
executables that call the new CLI, preserving backward compatibility for
existing users and documentation.

### 4.3 Component responsibilities

- **common.sh** — shared: argument parsing, color output, path-safe game
  search, SHA256 verification, architecture check, quarantine removal,
  ad-hoc resigning, exit-code conventions.
- **install.sh** — orchestrates the full pipeline (see §5).
- **uninstall.sh** — restores backup, removes cream_api.ini, cleans up.
- **status.sh** — read-only diagnostic: game found? patch installed? backup
  present? installed dylib matches expected SHA256?

## 5. dylib install pipeline (the stability fix)

Per install, in order, each step aborts on failure with a clear message:

1. **Integrity check** — SHA256 of `vendor/creamapi/libsteam_api.dylib` must
   match `vendor/creamapi/VERSION.txt`. Exit code 4 on mismatch (corruption/
   tampering).
2. **Architecture check** — `lipo -info` confirms the dylib contains both
   `arm64` and `x86_64`. Warn (do not fail) if only one slice is present.
3. **Locate game & target dylib** — path-safe search (see §6.1).
4. **Guard against double-patch** — if the found `libsteam_api.dylib` already
   matches our CreamAPI SHA256, the patch is already installed → report and
   exit cleanly (idempotent).
5. **Backup original** — copy to `libsteam_api_o.dylib`, but only if that
   backup does not already exist (never overwrite a good backup).
6. **Copy** — CreamAPI dylib + generated `cream_api.ini` (appid substituted
   from the game conf) into the Frameworks dir.
7. **Permissions** — `chmod 755` dylib, `chmod 644` ini.
8. **Remove quarantine** — `xattr -dr com.apple.quarantine` on copied files.
9. **Ad-hoc resign** — `codesign --force -s -` on the copied dylib. Removes
   the leaky `Identifier=config-arm64.out` build artifact and makes the
   signature consistent so macOS loads the library inside the game process.
10. **Verify result** — re-check SHA256 of installed dylib; confirm
    `libsteam_api_o.dylib` backup present.

`status` re-runs steps 1–2 and the installed-dylib check, so users can
detect when Steam's "Verify Integrity of Game Files" silently reverted the
patch.

## 6. Defects fixed

### 6.1 Path-with-spaces bug (critical)

Current code:
```bash
for expanded_path in $path; do     # $path unquoted → splits on spaces
```
Fails for `/Volumes/My Book/SteamLibrary/...`.

New approach: a single `find` with properly quoted `-path` patterns; custom
paths passed via `--path` are used verbatim. No unquoted variable expansion
in word-splitting contexts.

### 6.2 Error handling (critical)

All scripts: `set -euo pipefail`. Every `cp`/`find`/`codesign` result is
checked. No silent "success" after a failed copy.

### 6.3 ShellCheck compliance

`read -r` (fixes SC2162). All other shellcheck warnings resolved. CI enforces
zero shellcheck findings.

### 6.4 Translation desync (critical docs)

`README_DE/ES/FR/ZH` still claim "single-player only, multiplayer NOT
supported" and reference Goldberg Emulator — directly contradicting v3.0+
(CreamAPI supports multiplayer). All five translations are rewritten to
match the primary README's v4.0 content and structure.

### 6.5 Platform table misinformation

Current README claims Apple Silicon needs Rosetta 2. False — the shipped dylib
is a fat binary with a **native arm64** slice. Table corrected: Apple Silicon
= Native (arm64).

### 6.6 DLC count mismatch

README said "100+ DLCs" while ini had 89. Replaced by the `unlockall` strategy
which has no fixed count. No stale number in docs.

### 6.7 Missing legal/ban warning

Current README omits the reality that this violates Steam/Paradox EULA and
can lead to account ban. Primary README gains a clear, honest warning.

### 6.8 Leaked build path & ad-hoc identifier

`otool -L` shows `/Users/Shared/repos/cream_api_clean/...` and codesign
identifier `config-arm64.out`. The path is in the dylib itself (upstream
build artifact, not fixable without rebuilding) but is documented honestly;
the identifier is normalized by our ad-hoc resign in step 9.

## 7. cream_api.ini generation

Template `config/cream_api.ini.tmpl`:
```ini
[steam]
appid = __APPID__
unlockall = true
orgapi = libsteam_api_o.dylib
forceoffline = false

[steam_misc]
disableuserinterface = false

[dlc]
```
At install, `__APPID__` is substituted from the game conf; the result is
written next to the dylib. Empty `[dlc]` per unlockall contract.

- **README.md** — v4.0, honest EULA/ban warning, corrected platform table
  (native arm64), no stale DLC count, FAQ adds: "Steam Verify Files reverted

## 9. CI (GitHub Actions)

- **shellcheck.yml** — lint every `.sh` on push/PR; must pass clean.
- **links.yml** — validate all markdown links across 6 READMEs (lychee or
  markdown-link-check).
- **verify-dylib.yml** — assert the committed dylib's SHA256 equals the value
  in `vendor/creamapi/VERSION.txt` (detect corruption/tampering in-repo).

## 10. Testing strategy

- **Static:** shellcheck (zero warnings) enforced in CI.
- **Smoke:** `./unlocker status eu4` is non-mutating and exercises search +
  integrity + installed-state checks without touching the game install.
- **Manual checklist** in CONTRIBUTING.md: install → status (patched) →
  uninstall → status (clean), on both Intel and Apple Silicon, with a
  path containing a space.
- **Out of scope (YAGNI):** bats unit tests — shellcheck + smoke + CI is
  sufficient for a project this size.

## 11. Explicit non-goals (YAGNI)

- Shipping configs for CK3/HOI4/Stellaris now (architecture supports it;
  add when there is a concrete request).
- Building CreamAPI from source (it is closed-source).
- Full `.app` re-signing or repackaging (only the dylib).
- bats unit-test framework.
- A GUI.
- Auto-downloading DLC content files (the unlocker only changes ownership
  state; content must already be present).

## 12. Risks

| Risk | Mitigation |
|---|---|
| `unlockall` needs Steam online | Stated requirement (legit base game); document clearly |
| Steam ban / EULA violation | Honest warning in README; not hidden |
| AV false positives | VirusTotal 0/64 documented in README + SECURITY.md |
| codesign/xattr need Xcode CLT | Detect missing tools, print install hint (`xcode-select --install`) |
| Steam Verify reverts patch | `status` detects it; FAQ explains |
| Bash portability (macOS bash 3.2) | Avoid bash 4+ features (no associative arrays); use POSIX where simple |

## 13. Success criteria

1. `./unlocker status eu4` works read-only and reports patch/backup state.
2. `install` then `uninstall` leaves the game dir byte-identical to before
   (verified by SHA256 of the original dylib).
3. Install works on a path containing spaces.
4. shellcheck passes clean on all scripts.
5. All 6 READMEs contain no Goldberg references, no single-player-only
   warning, no `YOUR_USERNAME` placeholder.
6. `vendor/creamapi/VERSION.txt` SHA256 matches the committed dylib.
7. CI green on shellcheck + links + dylib verification.
